#### Basic usage

Default loader without customization. Useful for simple cases where a generic loading state is needed.

<Flex gap="m" direction="column">
  <Loader />
  <Loader view="bars" size="l" />
</Flex>

```tsx
function Demo() {
  return (
    <Flex gap="m" direction="column">
      <Loader />
      <Loader view="bars" size="l" />
    </Flex>
  );
}
```

#### Visual styles

The `view` prop defines the loader’s animation style. Choose between `oval`, `bars`, and `dots` depending on the context.

<Flex gap="m" direction="column">
  <Loader view="oval" />
  <Loader view="bars" />
  <Loader view="dots" />
</Flex>

```tsx
<Flex gap="m" direction="column">
  <Loader view="oval" />
  <Loader view="bars" />
  <Loader view="dots" />
</Flex>
```

#### Sizing options

The `size` prop accepts both design tokens (`xs`, `s`, `m`, `l`) and custom numeric values. Use tokens for consistency with the design system, and numbers for precise control.

<Flex gap="m" direction="column">
  <Loader size="xs" />
  <Loader size="s" />
  <Loader size="m" />
  <Loader size="l" />
  <Loader size={60} />
</Flex>

#### Best practices

##### When to use
  - Initial page load — during app bootstrap
  - Data fetching — indicate async operations (API calls, DB queries)
  - Form submissions — show processing state after user action
  - Content transitions — route changes, tab switching
  - Background processes — long-running tasks (exports, calculations)

##### When to not use
  - Instant actions — avoid loaders for operations under 300 ms
  - As decoration — loaders should always represent actual activity
  - Static content — don’t display loaders permanently without state
