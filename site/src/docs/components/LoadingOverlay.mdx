#### Basic usage

This example shows the simplest case — when loading is true, an overlay covers the content and displays a default spinner.

<LoadingOverlayBase />

```tsx
const LoadingOverlayBase = () => {
  const [loading, setLoading] = useState(false);
  return (
    <Flex direction="column" gap="l">
      <LoadingOverlay loading={loading}>
        <Flex direction="column" gap="m">
          <Flex gap="m">
            <TextField label="First name" />
            <TextField label="Last name" />
          </Flex>
          <TextField label="Email" />
          <TextField label="Password" />
        </Flex>
      </LoadingOverlay>
      <Flex justify="center">
        <Button onClick={() => setLoading((prev) => !prev)}>
          Toggle overlay
        </Button>
      </Flex>
    </Flex>
  );
};
```

#### Custom loader variant

loaderVariant allows you to select a different visual style (e.g., dots, bar, oval) while preserving overlay behavior.

#### Best practices

  - Do not put LoadingOverlay on the entire page if only a part of the UI is loading. Wrapping only relevant areas improves UX.
  - Show descriptive labels (“Uploading files…”, “Checking payment…”) to reduce uncertainty — especially for long operations.
  - While dimming is useful for modals or blocking actions, keep it off for small components to preserve readability.
  - If the overlay appears behind sticky headers or nested positioned blocks, increase `zIndex` to ensure proper stacking.
  - Use the same loader style across similar UI surfaces (e.g., all cards use “dots”, global overlay uses “oval”).
  - The overlay visually blocks content, but the container might still receive pointer events. Make sure the overlay layer covers interactions.
  - If possible, combine with fade transitions when showing/hiding overlays for a more polished experience.
